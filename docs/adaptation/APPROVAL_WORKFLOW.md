# Approval Workflow

When the adaptive optimizer identifies a beneficial ranking change for a family operating in `recommend_only` or `auto_apply_low_risk` (for non-qualifying families) mode, it creates an `AdaptationRecommendation`. This recommendation must pass through the approval workflow before it can be applied.

## Recommendation Creation

Recommendations are generated by `AdaptationRecommendationService.generateRecommendation()` when:

1. A plateau signal is detected (moderate or severe severity in `auto_apply_low_risk` mode; any severity in `recommend_only` mode).
2. The optimizer has computed a new candidate ranking that differs from the current state.
3. The adaptive mode does not permit automatic application.

Each recommendation includes:
- **familyKey** -- the execution family affected.
- **recommendedRanking** -- the proposed candidate ordering.
- **evidence** -- a human-readable summary of plateau indicators, trend data, and scoring rationale.
- **status** -- initially `pending`.

Once created, the recommendation is submitted to the `AdaptationApprovalService` via `submitForApproval()`, which creates a corresponding `AdaptationApproval` record.

## Approval States

| Status | Description |
|---|---|
| `pending` | Awaiting human decision. The approval is active and visible in the approval queue. |
| `approved` | Accepted by a human actor. The ranking change may now be applied. |
| `rejected` | Declined by a human actor. The ranking change will not be applied. |
| `expired` | The approval timed out without a decision. Default expiry is 24 hours. Expired approvals cannot be retroactively approved. |
| `superseded` | A newer recommendation for the same family was submitted, making this one obsolete. Superseded approvals cannot be approved. |

## State Transitions

```
pending  -->  approved   (via approve action)
pending  -->  rejected   (via reject action)
pending  -->  expired    (via expireStale sweep)
pending  -->  superseded (when a newer recommendation is submitted for the same family)
```

Terminal states (`approved`, `rejected`, `expired`, `superseded`) cannot transition to any other state.

## Approve Flow

1. Operator navigates to the approval queue in the admin web UI or calls the API endpoint.
2. Operator reviews the recommendation detail: evidence summary, current vs. proposed ranking, family performance data.
3. Operator clicks "Approve" and optionally provides a reason.
4. `AdaptationApprovalService.approve(id, actor, reason)` is called.
5. The approval status is set to `approved`, and the `decidedAt`, `decidedBy`, and `reason` fields are populated.
6. An `approval_approved` audit event is emitted.
7. The system applies the recommended ranking change to the family's optimizer state.

## Reject Flow

1. Operator reviews the recommendation and determines it should not be applied.
2. Operator clicks "Reject" and provides a reason (recommended but not required).
3. `AdaptationApprovalService.reject(id, actor, reason)` is called.
4. The approval status is set to `rejected`.
5. An `approval_rejected` audit event is emitted.
6. The family's ranking remains unchanged.

## Expiry

The `AdaptationApprovalService.expireStale()` method is called periodically (typically by a worker job) to expire pending approvals that have passed their `expiresAt` timestamp. Default expiry is 24 hours from submission.

Expired approvals emit an `approval_expired` audit event. They cannot be retroactively approved or rejected.

## Audit Implications

Every state transition emits an `ApprovalAuditEvent` via the configured `ApprovalAuditEmitter`:

| Event Type | Trigger |
|---|---|
| `approval_submitted` | A new recommendation enters the approval queue. |
| `approval_approved` | An operator approves a pending recommendation. |
| `approval_rejected` | An operator rejects a pending recommendation. |
| `approval_expired` | A pending recommendation times out. |

Audit events include the `approvalId`, `familyKey`, `actor` (when applicable), `reason` (when provided), and `timestamp`. These events are persisted in the audit log and are queryable via the audit API.

## API Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/adaptation/approvals` | List approvals (filterable by `status`, `familyKey`). |
| `GET` | `/adaptation/approvals/:id` | Get approval detail. |
| `POST` | `/adaptation/approvals/:id/approve` | Approve a pending recommendation. Body: `{ reason?: string }` |
| `POST` | `/adaptation/approvals/:id/reject` | Reject a pending recommendation. Body: `{ reason?: string }` |
